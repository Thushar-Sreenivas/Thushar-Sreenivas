# Pitfalls Research

**Domain:** GitHub Profile README / Markdown SVGs & Animations
**Researched:** 2026-04-12
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: Interactive SVG Delusion (The "Clickable Spells" Trap)

**What goes wrong:**
Attempting to embed CSS `:hover` states, JavaScript, or `<a>` link tags *inside* the SVG for interactive easter eggs. The interactions work locally in the browser, but fail completely when pushed to GitHub.

**Why it happens:**
GitHub's pipeline strictly requires embedding SVGs via `<img src="art.svg">` or `<picture>` tags inside the Markdown file. Browsers deliberately disable all interactive features, scripts, and pointer events inside SVGs loaded via the `<img>` tag for security and isolation reasons. 

**How to avoid:**
Do not put interactivity *inside* the SVG. If you want a clickable link, wrap the entire image tag in the Markdown (`[![Alt text](art.svg)](link)`). For hover animations, rely purely on autonomous looping animations (e.g., CSS `@keyframes` that run continuously) or SMIL `<animate>`.

**Warning signs:**
The SVG file has `:hover` pseudo-classes or `<a href="...">` tags inside its source code. It works perfectly when opening `file://.../art.svg` but breaks in an HTML test page using `<img src="art.svg">`.

**Phase to address:**
Phase 1 (Complex SVG Generation)

### Pitfall 2: GitHub Camo Proxy Cache Limits (The "Stunning GIF" Trap)

**What goes wrong:**
The new algorithmic art background GIF refuses to load or stops animating after pushing to GitHub, displaying a broken image icon or a statically rendered first frame.

**Why it happens:**
GitHub routes all repository images through its Camo proxy caching service. Camo has strict timeouts (~4 seconds to fetch) and file size limitations (traditionally causing massive performance issues or complete rendering failure for files >5-10MB). High-framerate algorithmic art easily hits these ceilings.

**How to avoid:**
Instead of generating a massive animated GIF for the generative art, leverage GitHub's native support for looping, autoplaying `.mp4` video. Render the algorithmic art to WebM or MP4 instead of GIF. This reduces file size by ~90%, bypasses the Camo 256-color banding, and guarantees smooth playback without crashing mobile clients.

**Warning signs:**
The generated GIF exceeds 5MB, or tests using `algorithmic-art` result in thousands of frames or high resolutions (>800px width).

**Phase to address:**
Phase 2 (Algorithmic Art Integration)

### Pitfall 3: External Font Blocking in SVGs (The "Frieren Aesthetic" Trap)

**What goes wrong:**
The "Frieren" themed SVGs (e.g., generated by Satori) render in Times New Roman or a generic Sans-serif font when viewed on the GitHub profile.

**Why it happens:**
GitHub's Content Security Policy (CSP) blocks external resource loading (like `@import url('https://fonts.googleapis.com...')` or `<link href="...">`) inside rendered SVGs. If the font isn't installed locally on the viewer's machine, the browser falls back to system fonts, breaking the aesthetic.

**How to avoid:**
Text inside SVGs MUST be converted to raw vector `<path>` elements before saving. If using Satori, use the `load-ttf` and font embedding options to ensure text is rendered as SVG geometry, preventing any reliance on external font files.

**Warning signs:**
SVG file contains `<text>` nodes instead of `<path>` nodes. No Base64 embedded fonts are present in the `<style>` block.

**Phase to address:**
Phase 1 (Complex SVG Generation)

---

### Pitfall 4: Stale Dynamic SVGs (The "Cron Job Caching" Trap)

**What goes wrong:**
A GitHub Actions cron job updates the "Grimoire" data SVG, but profile visitors still see yesterday's data.

**Why it happens:**
GitHub's Camo proxy caches images aggressively based on the specific URL string. If the SVG URL is the same (e.g., `stats.svg`), Camo doesn't immediately fetch the new version, leading to permanently stale data.

**How to avoid:**
Implement a cache-busting technique. Have the cron job modify the `README.md` to append a commit hash or timestamp query parameter to the image source (e.g., `<img src="stats.svg?v=a1b2c3d">`), OR ensure the server providing the dynamic SVG sets `Cache-Control: no-cache, max-age=0, must-revalidate` headers. Since this project limits infrastructure to native GitHub features, appending the hash to the `README.md` via the build script is mandatory.

**Warning signs:**
The GitHub action succeeds, the SVG file is updated in the repo, but a hard-refresh on the README page still shows the old SVG.

**Phase to address:**
Phase 3 (Automation / Grimoire Updates)

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Using `<text>` elements instead of `<path>` for typography | Smaller SVG file size, easier to generate | Breaks aesthetic on 95% of devices due to missing custom fonts | Never acceptable for a highly-themed profile |
| Hardcoding `#0d1117` directly in the dark SVG | Perfect match for GitHub dark mode | Breaks entirely if GitHub ever shifts their theme color palette | Acceptable only if transparency is impossible |
| High framerate (30+ fps) GIF output | Silky smooth animations locally | Fails to load behind GitHub proxy; crashes mobile app rendering | Never acceptable; switch to MP4 or limit GIF to 10fps |

## Integration Gotchas

Common mistakes when connecting to external services.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| GitHub Markdown `<picture>` | Omitting explicit width/height on the fallback `<img>` | Define dimensions to prevent layout shifts/content jumping when dark theme swaps |
| Satori | Relying on remote URLs for images in the generated SVG | Base64-encode all image assets directly into the Satori template before rendering |
| Actions Cron | Not committing the updated `README.template.md` cache-buster | Modify both the SVG file *and* the referencing Markdown file during the action |

## Performance Traps

Patterns that work at small scale but fail as usage grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Massive animated GIFs | Profile page lags when scrolling on mobile devices | Use MP4 fallback or heavily compress/dither the GIF | GIF size > 3MB |
| Deeply nested SVG nodes | High CPU usage when rendering complex generative art | Flatten paths and optimize the algorithmic SVG output | SVG node count > 10,000 |
| Overlapping transparency in SVG | Browser struggles to composite the SVG (especially on Safari) | Bake transparency into flat colors where possible | Heavy alpha channels used |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Using `foreignObject` or `<script>` tags in SVG | SVG gets entirely stripped by GitHub Markdown sanitizer, rendering a blank block | Use strict subset of SVG standard; avoid all JavaScript inside the SVG |
| Injecting dynamic Github stats via untrusted API | XSS or broken layout if the API changes payload | Verify and sanitize data schemas from GitHub API before passing to Satori |

## UX Pitfalls

Common user experience mistakes in this domain.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Missing light mode fallbacks | Light mode users see an illegible dark block | Use GitHub's `#gh-dark-mode-only` and `#gh-light-mode-only` classes to handle switching |
| Autoplaying intense animations | Distracting to users, causes motion sickness | Ensure generative art is subtle, ambient, and non-intrusive (slow moving) |
| Tiny font sizes in rendered SVGs | Unreadable on mobile screens | Ensure Satori templates use responsive scaling or larger base font sizes |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **Interactive Elements:** Often missing clickability — verify the wrapping `<a>` tag is at the Markdown level, not SVG level.
- [ ] **Font Rendering:** Often missing correct styling — verify by testing the raw SVG on a fresh device without the font installed.
- [ ] **Generative Animation:** Often missing performance limits — verify the file size is under 5MB and frame rate is strictly controlled.
- [ ] **Data Sync:** Often missing cache invalidation — verify the markdown contains a unique query string appended to the image URL.

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Camo GIF proxy block | LOW | Replace `.gif` asset with `.mp4`, update markdown links |
| Font fails to load in SVG | MEDIUM | Switch SVG generation pipeline to use `text-to-path` tools (e.g., opentype.js) |
| Cached stale stats image | LOW | Force a commit that updates the cache-buster query string parameter |
